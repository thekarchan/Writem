# Writem Markdown 编辑器功能需求文档（V1）

## 1. 产品目标

打造一个以写作体验优先的 Markdown 编辑器，兼顾：

- 普通写作者的低门槛输入
- 技术用户对 Markdown 原文的控制感
- 博客作者对本地文件、Frontmatter、静态站点发布流程的适配

### 核心目标

1. 让用户可以在单栏沉浸环境中高效写作
2. 降低 Markdown 语法使用门槛
3. 保持 `.md` 文件和目录结构原生可控
4. 解决图片、表格、Frontmatter、发布检查等真实痛点
5. 支持从写作到发布前检查的完整闭环

## 2. 产品范围

### 2.1 V1 必做范围

1. 单栏沉浸式编辑
2. 增强型 Markdown 显示
3. 基础格式快捷输入
4. 图片插入与资源管理
5. 表格可视化编辑
6. 代码块增强
7. Frontmatter 可视化编辑
8. 大纲导航
9. 块级折叠
10. 发布前检查
11. 本地文件读写与自动保存
12. 导出能力（HTML / PDF / Markdown 原文）

### 2.2 V1 暂不纳入

1. 实时多人协作
2. 云同步
3. 插件市场
4. AI 写作能力
5. Git 内置提交管理
6. 数据库式双向链接系统
7. Notion 式块数据库

## 3. 用户角色

### 3.1 普通写作者

需求：

- 不熟悉 Markdown
- 想快速排版
- 想少记语法

### 3.2 技术用户

需求：

- 保留 Markdown 原文控制
- 代码块体验专业
- 文件可迁移、可版本管理

### 3.3 博客作者

需求：

- 本地目录写作
- Frontmatter 配置简单
- 图片路径规范
- 发布前检查完整

## 4. 功能需求

### 4.1 单栏沉浸式编辑

#### 功能说明

编辑器默认采用单栏写作布局，聚焦正文内容，减少左右预览切换带来的视线干扰。

#### 功能点

- 默认进入单栏编辑模式
- 行宽可配置
- 支持聚焦当前段落 / 当前行高亮
- 支持阅读模式、源码模式切换
- 支持浅色 / 深色主题

#### 交付标准

- 用户首次打开文档时默认进入单栏编辑态
- 切换模式响应时间 `< 200ms`
- 光标位置在模式切换后保持不变或误差不超过 1 行

#### 验收标准

- 写作态下无双栏强依赖
- 视觉层级清晰，不影响连续输入
- 小屏设备下布局不破坏

#### 测试用例

用例 1：默认进入单栏编辑

- 前置条件：打开任意 md 文件
- 操作：进入编辑器
- 预期结果：默认显示单栏正文编辑区，无右侧预览强制展示

用例 2：模式切换

- 操作：从写作态切换到阅读态，再切换到源码态
- 预期结果：内容不丢失，滚动位置基本保持一致，显示正确

### 4.2 增强型 Markdown 显示

#### 功能说明

在不隐藏 Markdown 本质的前提下，对标题、列表、引用、代码块等进行增强渲染。

#### 功能点

- 标题具备明显字号层级
- 列表具备缩进和间距
- 引用块左侧显示样式标识
- 代码块独立背景显示
- 语法符号弱化显示，但不彻底消失
- 光标进入当前块时，语法完整可见

#### 交付标准

- 支持 H1-H6、无序列表、有序列表、引用、分割线、粗体、斜体、行内代码、代码块基础增强显示
- 渲染正确率 `>= 95%`

#### 验收标准

- 用户可以明确区分不同块类型
- 不出现“看起来是富文本，实际编辑失控”的问题

#### 测试用例

用例 1：标题显示

- 输入：`# 标题1`
- 预期：显示为一级标题样式，`#` 保留但弱化

用例 2：引用块显示

- 输入：`> 这是一段引用`
- 预期：左侧出现引用视觉标记，文本样式区别于正文

用例 3：光标进入块

- 操作：将光标移入已渲染的标题行
- 预期：Markdown 语法符号显示完整，不影响编辑

### 4.3 基础格式快捷输入

#### 功能说明

提供快捷方式降低 Markdown 学习成本。

#### 功能点

- `/` 命令面板
- 选中文本后可一键加粗、斜体、链接、行内代码
- 输入 `- ` 自动转无序列表
- 输入 `1. ` 自动转有序列表
- 粘贴 URL 自动识别链接
- 快捷键支持常见格式操作

#### 交付标准

- `/` 命令支持至少：标题、列表、引用、代码块、表格、分割线、图片
- 常见快捷键与系统约定保持一致

#### 验收标准

- 新用户无需记忆完整语法，也能完成基本排版

#### 测试用例

用例 1：斜杠命令插入代码块

- 操作：输入 `/code`
- 预期：插入标准 Markdown 代码块模板

用例 2：选中文本加粗

- 操作：选中“测试文字”，点击加粗
- 预期：文本被包裹为 `**测试文字**`

用例 3：粘贴 URL

- 操作：粘贴一个完整网址
- 预期：自动生成链接格式或可点击识别

- 相对路径生成正确

- 用户无需手动复制、移动、重命名图片即可完成插图

用例 3：路径正确

- 预期：图片路径按规则生成，且为相对路径

### 5.2 稳定性

- 自动保存可靠
- 崩溃恢复可用
- 图片插入不丢文件

### 5.3 可用性

- 新用户 5 分钟内可完成标题、列表、插图、表格基本操作

### 5.4 兼容性

- 支持标准 Markdown
- Frontmatter 兼容主流静态站点

## 8. 风险点

### 8.1 富文本感与 Markdown 控制感冲突

风险：过度渲染后，用户失去源码可控感。  
处理：始终保留源码模式与语法可见性。

### 8.2 表格编辑复杂度高

风险：表格体验容易变成半吊子 spreadsheet。  
处理：V1 只做基础增删改和粘贴转换，不做复杂公式。

### 8.3 图片路径混乱

风险：跨目录移动导致图片失效。  
处理：统一资源目录策略，插入时就规范化。

### 8.4 长文性能

风险：增强渲染和大纲同步导致卡顿。  
处理：块级渲染、增量更新、按需折叠。
